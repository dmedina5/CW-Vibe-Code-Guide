Mental Model

Claude Code is not a chatbot; it's an agent in your terminal. Think of it as a very capable junior developer who needs clear direction and context.

🧠 How Claude Thinks
  • Reads files to understand your codebase
  • Plans before executing (especially in plan mode)
  • Asks permission before making changes
  • Maintains context within a session

The 20-Turn Cliff

Claude's reasoning quality can degrade after ~20 back-and-forth turns. The context gets polluted with false starts and corrections. When you notice confusion:

# Reset the session while keeping CLAUDE.md context
/reset

# Or start fresh with compacted context
/compact

Context Architecture: CLAUDE.md

CLAUDE.md is Claude's map to your project. It's automatically read at session start and provides persistent context.

Location

# Project root (recommended)
./CLAUDE.md

# User-level (for personal preferences)
~/.claude/CLAUDE.md

Example CLAUDE.md

# Project: CW Premium Calculator

## Overview
This is a TypeScript monorepo for insurance premium calculations.
We use pnpm workspaces, Vitest for testing, and Prisma for ORM.

## Build Commands
- `pnpm install` - Install dependencies
- `pnpm build` - Build all packages
- `pnpm test` - Run tests
- `pnpm lint` - Run linting

## Architecture
- /packages/core - Business logic
- /packages/api - Express REST API
- /packages/web - Next.js frontend

## Conventions
- Use explicit types, avoid `any`
- Tests go in `__tests__` folders
- Prefer functional patterns

## Important Files
- packages/core/src/premium.ts - Main calculation logic
- packages/api/src/routes/quote.ts - Quote API endpoint

4-Level Signal Framework

Structure your CLAUDE.md with different signal levels:

  1. Critical - Must-know info (build commands, architecture)
  2. Important - Project conventions, key files
  3. Contextual - Current sprint focus, recent changes
  4. Nice-to-have - Historical decisions, future plans

Permission System

Claude asks permission before potentially destructive actions. You can configure autonomy levels:

# Check current permissions
/permissions

# Run with full autonomy (careful!)
claude --dangerously-skip-permissions

# Alias for convenience
alias cc='claude --dangerously-skip-permissions'
⚠️ Danger Mode Warning

Only use --dangerously-skip-permissions on projects where you're comfortable with Claude making changes without asking.

Extending Claude: Commands & Skills

Claude can be extended with reusable prompt templates that save time and enforce consistency.

Slash Commands (Simple)

Markdown files that expand into prompts when you type /command-name:

# Location: ~/.claude/commands/ or .claude/commands/
# File: .claude/commands/review.md

---
name: review
description: Quick code review
---

Review this code for bugs, security issues, and style.

# Usage:
/review src/payment.ts

Skills (Powerful)

Full-featured modules with tool restrictions, model selection, and automatic invocation:

# Location: ~/.claude/skills/ or .claude/skills/
# File: .claude/skills/my-skill/SKILL.md

---
name: my-skill
description: What this skill does (helps auto-invoke)
allowed-tools: Grep, Read, Glob    # Restrict available tools
user-invocable: true               # Can use /my-skill
model: haiku                       # Use fast model for searches
---

# Detailed instructions for the skill...

# Usage:
/my-skill [arguments]

Mental Model

Gemini CLI, powered by Gemini 3, is your force multiplier. It excels at holding massive context (up to 2M tokens with Gemini 3 Pro) and delegating to specialized sub-agents.

⚡ How Gemini 3 Thinks
  • Can hold entire codebases in context (2M tokens)
  • Delegates to sub-agents for complex tasks
  • Has persistent memory across sessions
  • Deep Think mode for complex reasoning
  • Optimized for breadth over depth

Gemini 3 Model Variants

  • Gemini 3 Pro - Maximum context (2M tokens), best reasoning
  • Gemini 3 Flash - Fast responses (1M tokens), everyday tasks
  • Gemini 3 Deep Think - Extended reasoning for complex problems

Sub-Agent Delegation

When you give Gemini a complex task, it may spawn sub-agents:

  • codebase_investigator - Maps dependencies and structure
  • skill-creator - Learns new tools on the fly
  • write_todos - Tracks multi-step progress

Leveraging Large Context

Gemini 3's massive context window is its superpower - up to 2M tokens with Gemini 3 Pro. Use it for:

  • Loading entire modules at once
  • Cross-file refactoring
  • Understanding legacy systems
  • Batch documentation generation
  • Deep Think mode for complex architectural analysis
# Reference files directly with @
> @src/modules/auth.ts @src/modules/user.ts
> "How do these modules interact?"

# Load entire directories
> "Read all files in packages/core/src"

# Use Deep Think for complex questions
> /model gemini-3-deep-think
> "Analyze the entire authentication flow and identify security concerns"

Persistent Memory

Unlike Claude, Gemini has built-in persistent memory across sessions:

# Save a fact for all future sessions
> "Remember: I always use 4-space indentation and prefer pytest over unittest"

# Gemini uses save_memory tool internally

# View saved memories
> "What do you remember about my preferences?"
💭 Memory Best Practices
  • Save coding style preferences
  • Save frequently-used commands
  • Save project-specific conventions
  • Don't save sensitive information

Understanding Markdown Files (.md)

Markdown files are the lingua franca of vibe coding. They're plain text files with simple formatting that both humans and AI agents can easily read and write.

📄 What is Markdown?

Markdown (.md files) is a lightweight markup language that uses simple symbols for formatting. It's readable as plain text but renders beautifully in tools like GitHub, VS Code, and documentation sites.

Why Markdown Matters for Vibe Coding

AI coding agents are trained extensively on markdown. This makes .md files the ideal format for:

  • Configuration - CLAUDE.md and GEMINI.md tell agents about your project
  • Planning - PLAN.md files for multi-step implementation strategies
  • Handoffs - HANDOFF.md for context transfer between agents
  • Documentation - README.md, API docs, guides
  • Task Lists - TODO.md with checkboxes agents can update
💡 Key Insight

When you write instructions in a .md file, AI agents parse them more reliably than other formats. The structured nature of markdown (headers, lists, code blocks) helps agents understand hierarchy and intent.

Basic Markdown Syntax

# Heading 1
## Heading 2
### Heading 3

**bold text**
*italic text*
`inline code`

- Bullet point
- Another bullet
  - Nested bullet

1. Numbered list
2. Second item

```javascript
// Code block with syntax highlighting
const example = "Hello World";
```

[Link text](https://example.com)

> Blockquote for callouts

| Table | Header |
|-------|--------|
| Cell  | Cell   |

- [ ] Unchecked task
- [x] Completed task

Essential Markdown Files for Vibe Coding

File Purpose Who Reads It
CLAUDE.md Project context for Claude Code Claude (auto-loaded)
GEMINI.md Project context for Gemini CLI Gemini (auto-loaded)
README.md Project overview and setup Humans + AI
PLAN.md Implementation strategy AI agents executing tasks
HANDOFF.md Context for agent switching Next AI agent in workflow
TODO.md Task tracking with checkboxes AI agents (can update)

Why AI Agents Love Markdown

  1. Structured but flexible - Headers create clear sections without rigid schemas
  2. Code-friendly - Code blocks preserve formatting and enable syntax highlighting
  3. Human-readable - You can read/edit markdown files directly
  4. Version control friendly - Plain text diffs cleanly in git
  5. Universal support - Every code editor, IDE, and platform supports markdown
✅ Pro Tip

When asking an AI agent to create a plan or document findings, explicitly ask for markdown format: "Write your analysis to ANALYSIS.md". This creates artifacts you can review, edit, and pass to other agents.

Context Hygiene

Both tools perform better with clean, focused context. Avoid polluting the conversation with irrelevant information.

🧹 Context Hygiene Rules
  1. Start fresh for new tasks
  2. Be specific about what you need
  3. Add relevant files explicitly
  4. Clear context when switching tasks

Effective Prompt Patterns

The Task Pattern

# Bad: Vague request
"Help me with the login"

# Good: Specific task
"In src/auth/login.ts, the login function throws an error
when the user doesn't exist. Instead of throwing, it should
return { success: false, error: 'USER_NOT_FOUND' }"

The Context Pattern

# Provide context before asking
"I'm working on the premium calculation module.
The current implementation uses a simple multiplier,
but we need to add tiered pricing based on coverage amount.

Here's the current code: [paste or reference file]

Please update it to use tiered pricing where:
- $0-50k: 1.0x multiplier
- $50k-100k: 0.95x multiplier
- $100k+: 0.90x multiplier"

The Checklist Pattern

"I need to add user authentication. Please:

1. Create a new auth middleware in src/middleware/
2. Add login/logout routes to src/routes/auth.ts
3. Create a User model if it doesn't exist
4. Add JWT token handling
5. Update the API documentation

After each step, show me what you did and wait for approval."

File Reference Patterns

Pattern Claude Code Gemini CLI
Add file to context /add path/to/file.ts @path/to/file.ts
Read file "Read src/app.ts" @src/app.ts
Multiple files /add src/*.ts @src/*.ts

Autonomy Tradeoffs

Both tools offer different autonomy levels. Choose based on your trust and task complexity:

🔒 High Oversight

When: New codebases, critical code, learning the tools

  • Review each change
  • Approve file operations
  • Step-by-step execution

🔓 Low Oversight

When: Familiar codebase, repetitive tasks, trusted patterns

  • Batch operations
  • Auto-approve reads
  • Let agent iterate
