üéØ Executive Summary

  • Claude Code is not ChatGPT with file access. It's an autonomous agent with tools, permissions, and execution authority inside a finite context window. Treating it like a chatbot causes most failures.
  • Context orchestration is the game. Every file, instruction, and conversation turn competes for attention. Clean context lets mediocre prompts succeed; more context is often worse.
  • Terminal-first is the point. Your shell gives Claude access to CLI tools, scripts, CI hooks, and automation that IDE assistants can't touch.
  • Autonomy is a dial. Permission prompts maximize safety but destroy flow. Custom allowlists or full autonomous mode (with version control as your safety net) restore agentic value.
  • Performance degrades after ~15-20 turns. This is a hard limit. Reset context, decompose tasks, or externalize state instead of pushing through.
  • CLAUDE.md should be minimal. Build commands, critical paths, non-obvious constraints. Bloated files trigger Chekhov's gun‚Äîmodels try to use everything they're given.
  • At Cover Whale: Claude Code excels at data migrations, validation scripts, API integrations, and the repetitive automation tasks common in insurance operations.

The Governing Rule: If you don't control context, you don't control outcomes. Claude Code rewards system design, not clever prompting.

üß† The Mental Model

All Levels

Most people approach Claude Code like ChatGPT with file access: type request, get answer, repeat. This guarantees mediocrity.

Claude Code is an autonomous agent operating in your codebase with a constrained tool set, a limited context window, and permission gates you control. Understanding this shift unlocks its full potential.

The Core Thesis: Context Orchestration

The quality of your outputs depends on how well you curate the information space Claude operates in. Every file reference, CLAUDE.md instruction, conversation turn, and external tool connection shapes the context window Claude reasons within.

Most users optimize their prompts. Power users optimize their context.

Terminal-First Is a Feature

The terminal interface feels like regression after years of IDE integrations. It's not‚Äîit's a deliberate design choice that unlocks capabilities IDE integrations can't match:

  • Full environment access: Claude inherits your shell, PATH, and tools‚Äîgh for GitHub, jq for JSON, ripgrep for search, docker for containers, and your custom scripts.
  • Scriptability: Headless mode (-p flag) lets you pipe Claude into CI pipelines, pre-commit hooks, and automation.
When to Use Claude Code vs IDE Assistants
‚úì

Claude Code Excels At

Complex multi-file refactors, migrations, debugging distributed issues, CI/CD automation, codebase-wide reasoning, headless batch operations, data transformation scripts

‚Üí

Consider IDE Tools For

Quick edits, tab completions, surgical single-file changes, inline suggestions during active coding

The Permission/Autonomy Tradeoff

Out of the box, Claude Code asks permission for almost everything. This is safety-first design for an agent executing code on your machine‚Äîbut it creates a workflow problem.

Fractured vs Recursive Logic Flow
‚è∏

Default Mode (Fractured)

Plan ‚Üí Wait for permission ‚Üí Execute ‚Üí Wait again ‚Üí Continue

Each pause breaks Claude's reasoning chain

‚Üª

Autonomous Mode (Recursive)

Plan ‚Üí Execute ‚Üí Observe ‚Üí Adjust ‚Üí Continue

Continuous feedback loop enables true agentic behavior

üí° Practical Recommendation

Create an alias: alias cc='claude --dangerously-skip-permissions'. Work in version-controlled repos, commit often, and revert when needed. The productivity gain is substantial‚Äîjust make sure you're in a git repo!

‚öôÔ∏è The Foundational Setup

Intermediate

Configuration decisions shape everything before you write a single prompt.

CLAUDE.md: Your Persistent Context

This file loads automatically at session start. Instructions here get followed more reliably than chat inputs.

üìÅ File Hierarchy

~/.claude/CLAUDE.md ‚Üí Applies to all your projects globally
./CLAUDE.md ‚Üí Project-specific, can be shared with team via git

What belongs in CLAUDE.md:

  • Build/test commands: "Run npm test before committing." "Use docker compose up for local dev."
  • Approved operations: git commands (add, commit, push) and build scripts to skip permission prompts.
  • Non-obvious constraints: "Never modify legacy_auth.py." "API changes require migration scripts."
  • Project context: Brief description of what the project does, key files, and architecture decisions.

Example CLAUDE.md for a Cover Whale Project:

# Project: Premium Validation Tool

## Overview
Python-based validation tool for insurance premium calculations.
Compares Solartis API responses against expected rates.

## Commands
- Run tests: `pytest tests/ -v`
- Format code: `black . && isort .`
- Type check: `mypy src/`

## Constraints
- Never hardcode API keys - use environment variables
- All rate calculations must include state-specific tax logic
- Changes to core validation logic require review

## Key Files
- src/validators/premium.py - Main validation logic
- src/rates/state_taxes.py - State tax calculations
- config/carriers/ - Carrier-specific rate configurations

Custom Slash Commands

Put markdown files in .claude/commands/ to create reusable prompt templates available as /project:filename.

Example: A fix-issue.md command that takes an issue number, runs gh issue view, searches for relevant files, implements fixes, writes tests, and commits.

üí° Claude-Specific Tips & Tricks

All Levels

Since you're literally asking Claude to help you use Claude Code, here are some insider tips that work particularly well.

üéØ

Use "Think Step by Step"

For complex tasks, explicitly ask Claude to think through the problem. "Think step by step about how to refactor this authentication flow, then propose a plan before coding."

üìù

Request Explanations First

Before asking Claude to modify code, ask it to explain what the code currently does. This primes better context and catches misunderstandings early.

üîç

Be Specific About Scope

"Only modify the calculate_premium function" or "Don't change any files in /legacy" prevents Claude from making well-intentioned but unwanted changes.

‚úÖ

Use Verification Steps

End requests with "After making changes, run the tests and show me the output." This creates a feedback loop that catches errors immediately.

üßπ

Reset Liberally

If Claude seems confused or is repeating itself, /reset and start fresh. Don't try to "fix" a corrupted context‚Äîit rarely works.

üìä

Specify Output Format

"Output the results as a CSV" or "Format this as a markdown table" gives Claude clear direction on structure, not just content.

Prompt Patterns That Work

üîß The Explore-Plan-Execute Pattern

Instead of "Build me X", try this three-step approach:

  1. "Read the files in src/validation/ and explain how the validation pipeline works. Don't write any code yet."
  2. "Now propose how you'd add support for [new feature]. List the files you'd modify and why."
  3. "That looks good. Implement it, then run the tests."
üêã The Constraint-First Pattern

Lead with constraints, not requirements:

  • ‚ùå "Write a function to calculate premium taxes"
  • ‚úÖ "Write a function to calculate premium taxes. It must: (1) handle all 50 US states, (2) use the rates in config/tax_rates.json, (3) return a dict with breakdown by tax type, (4) include type hints and docstrings."

Extended Thinking for Complex Tasks

Claude has access to extended thinking capabilities. For complex architectural decisions or debugging sessions, you can explicitly request deeper analysis:

"This bug has been hard to track down. Take your time and think carefully 
about all the places where this race condition could occur. Consider the 
database transactions, the API calls, and the caching layer."

‚ö†Ô∏è Common Failure Modes

All Levels

Most complaints about Claude Code trace back to structural issues, not model limitations. Each has a mechanical fix.

Planning Failures

Premature Execution

Skips planning, dives straight into coding

‚Üí

Fix: Enforce plan ‚Üí approve ‚Üí code sequence. "Explain first, don't write code yet."

Over-Specification

500-word prompts anchoring to suboptimal approaches

‚Üí

Fix: Specify the what, be loose on the how. "Max 5 attempts per 15 min" beats implementation specs.

One-Shot Syndrome

Expecting complex tasks to work on first try

‚Üí

Fix: Treat outputs as drafts. Build in verification‚Äîtests, screenshots, behavior checks.

Context Failures

Context Contamination

Long sessions where everything bleeds together

‚Üí

Fix: Reset aggressively with /reset. Save state externally via commits and checklists.

20-Turn Cliff

Contradictions, repetition, hallucinated features

‚Üí

Fix: Break tasks into stages. Checkpoint with commits. Start fresh sessions after ~15 turns.

CLAUDE.md Bloat

300-line instruction files triggering Chekhov's gun

‚Üí

Fix: Keep it operational‚Äîcommands, constraints, stack. Nothing else.

Operational Failures

Permission Friction

Constantly waiting for approvals, breaking flow

‚Üí

Fix: Configure allowlists in CLAUDE.md or use --dangerously-skip-permissions with version control.

Ignoring Terminal Output

Missing errors, assuming success without verification

‚Üí

Fix: Check terminal output. Ask explicitly: "Did the tests pass? Show me any errors."

Writer-Reviewer Split

Use two Claude instances: one writes code, another reviews it. The context shift forces self-critique and prevents single-thread ratholing.

MCP: Model Context Protocol

MCP connects Claude to external tools. Without it, Claude is limited to filesystem and shell. With it, Claude can query databases, control browsers, post to Slack, and read from Notion. Configuration goes in ~/.claude/mcp.json.
